%% ============================================================
\section{Step 1: Basic Accelerator: Bias and ReLU in Hardware}
%% ============================================================

The HLS accelerator \texttt{Conv2D\_HW} mirrors the software 6-nested-loop structure, with
\texttt{input}/\texttt{output} and \texttt{filters}/\texttt{biases} split across two AXI master
bundles (HP0 and HP1) to achieve II=1. Bias addition and ReLU are folded into the
accumulation step, eliminating separate DDR passes. ReLU is parameterised to handle
all conv layers (ReLU=1) and the dense layer (ReLU=0).

%% ============================================================
\section{Step 2: Caching of Filter Coefficients}
%% ============================================================

Filter coefficients were re-fetched from DDR on every MAC operation. The fix copies
each filter into a local BRAM array (\texttt{localFilters[256*9]}) once per filter via
a pipelined burst, reducing DDR filter reads from $N_f \times H \times W \times C_i \times 9$
to $N_f \times C_i \times 9$. AXI burst length was raised to 256 beats with 16
outstanding requests to hide DDR latency.

%% ============================================================
\section{Step 3: Caching of Input Rows}
%% ============================================================

Three row buffers form a circular buffer: after the initial 3-row load, each y-iteration
fetches only 1 new row from DDR. \texttt{\#pragma HLS LOOP\_FLATTEN off} is applied to
the row-loading loops to avoid negative timing slack caused by large inter-channel stride
computations in CHW layout flattening the critical path beyond 10\,ns.

%% ============================================================
\section{Step 4: Parallelising Output Filters}
%% ============================================================

Computing $N$ output filters simultaneously is fully independent per input pixel. The
solution uses \texttt{localFilters[N][256*9]} with \texttt{ARRAY\_PARTITION complete dim=1}
and an unrolled inner loop to instantiate $N$ multipliers. Filter loading is interleaved
so all $N$ BRAMs are written from a single DDR read stream.

With N=16, scattered output writes became the bottleneck. An output row buffer
(\texttt{outRowBuf[16][256]}) accumulates one full row per filter in BRAM before
burst-writing to DDR, giving a 7.2$\times$ speedup on Conv 0.

%% ============================================================
\section{Step 5 (Optional): Row Reuse and Overlapping Transfers}
%% ============================================================

\subsection*{5a: 4-Buffer Prefetch}

A 4th row buffer eliminates the slot collision between prefetch and computation present
with 3 buffers. A side benefit: modulo-4 is a 2-bit mask vs.\ a full division circuit
for modulo-3, reducing pipeline iteration latency from 43 to 11 cycles (II=1 maintained).

\subsection*{5b: 2$\times$ X-position Parallelism}

Adjacent x-positions share the same filter coefficient. The x-loop steps by 2 with dual
accumulators, reading two input pixels and one filter value per cycle, doubling MAC
throughput without extra BRAM. Row buffers are partitioned cyclic with factor=2 for
conflict-free parallel access. DSP utilisation rose from 41\% to 63\%.

\subsection*{Negative Slack Fix}

Removing \texttt{LOOP\_FLATTEN off} caused negative timing slack. It was re-applied
selectively to row-loading and output-write loops while keeping the interleaved
filter-loading structure.

%% ============================================================
\section{Bonus: MaxPool Hardware Accelerator}
%% ============================================================

After Conv2D optimisations, Conv and MaxPool were equally bottlenecked (43.5\% and 43.1\%
of runtime). MaxPool 2$\times$2 stride 2 requires only comparisons (zero DSPs), making
it ideal for a lightweight accelerator. The HLS design reads two consecutive rows in a
single burst, applies a comparison tree at II=1, and burst-writes the output. Two
additional HP ports (HP2, HP3) handle MaxPool independently from Conv2D.
